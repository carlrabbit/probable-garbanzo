# Analyzer best practices

Creating a .NET analyzer is essentially writing code that runs every time your peer types a character. If it’s slow, the IDE feels sluggish, and the CI/CD pipeline grinds to a halt. Think of your analyzer as a background stagehand: if they’re doing their job right, you don’t notice them; if they’re clumsy, the whole show falls apart.

To keep your analyzers snappy in 2026, follow these performance-first best practices.

## 1. The "Bail Out Early" Pattern

The most expensive thing an analyzer can do is ask the compiler for "meaning" (semantics). The cheapest thing it can do is look at "shapes" (syntax).

**The Strategy:** Always perform syntactic checks before semantic ones.

- **Check the Kind:** If you're looking for InvocationExpression, check that first.
- **Check the Name:** If you only care about a method named Execute, check the string name of the identifier before calling GetSymbolInfo.

## 2. Infrastructure: Setting the Stage

Before writing a single line of logic, you must configure your analyzer to be a good citizen of the ecosystem.

### Enable Concurrent Execution

By default, analyzers run on a single thread. In modern multi-core environments, this is a waste.

```csharp
context.EnableConcurrentExecution();
```

*Note: Ensure your analyzer is stateless (thread-safe) before enabling this.*

### Ignore Generated Code

Analyzers should rarely run on code generated by Protobuf, EF Core, or Source Generators.

```csharp
context.ConfigureGeneratedCodeAnalysis(GeneratedCodeAnalysisFlags.None);
```

Analyzing thousands of lines of machine-generated code is the fastest way to spike CPU usage for zero benefit.

## 3. Registration Strategy: Knowing Your Costs

Not all "hooks" into the compiler are created equal. Use the most specific registration possible.

| Registration Type | Performance Cost | Best For... |
|---|---|---|
| **RegisterSyntaxNodeAction** | Low | Specific patterns (e.g., if statements). |
| **RegisterSymbolAction** | Medium | Checking members, types, or namespaces. |
| **RegisterOperationAction** | Medium | Cross-language logic (C# and VB.NET). |
| **RegisterSemanticModelAction** | High | Analyzing a whole file's logic at once. |
| **RegisterCompilationAction** | Very High | Global rules (avoid if possible). |

> **Pro-Tip:** Use IOperation (via RegisterOperationAction) when possible. It provides a higher-level, semantically bound view of the code that is often more efficient than manually stitching together syntax and symbols.

## 4. Memory & Allocations: The Silent Killers

Analyzers run thousands of times. Small allocations add up to massive Garbage Collection (GC) pressure.

- **Avoid LINQ:** In the hot path of an analyzer, foreach is your friend. LINQ creates hidden delegate allocations.
- **Use ImmutableArray:** Roslyn loves immutable collections. They are highly optimized for these workflows.
- **Cache Symbols in CompilationStart:** If you need to look up a common type (like System.Exception), do it once inside RegisterCompilationStartAction and pass it down to your nested node actions.

## 5. Summary Checklist

- [ ] Did I call EnableConcurrentExecution()?
- [ ] Did I exclude generated code?
- [ ] Am I checking node.IsKind(SyntaxKind...) before calling GetSymbolInfo?
- [ ] Did I avoid RegisterCompilationAction in favor of more granular hooks?
- [ ] Is my logic free of heavy LINQ queries?
